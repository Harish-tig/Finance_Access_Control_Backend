Project Overview

Finance Dashboard API is a backend service for managing financial records and generating analytics dashboards. It ships with a complete RBAC model, a dual-token auth system, an async Redis cache layer, and configurable rate limiting — all wired up automatically on first run.

🔐
Auth & Access
JWT access and refresh tokens. Three roles — Viewer, Analyst, Admin — control access at the route level via dependency injection.
📊
Financial Records
Create, query, update, and soft-delete income and expense entries. Full filter and pagination support on list endpoints.
⚡
Dashboard Analytics
Aggregated income, expenses, net balance, category totals, and monthly trends. Results cached in Redis with automatic invalidation on writes.
🛡️
Rate Limiting
Per-IP sliding-window rate limiter backed by Redis. Configurable request count and window duration via environment variables.
🌱
Auto-Seeded Admin
On first run, the database schema is created and a default admin account is seeded automatically — no manual migrations needed.
🔄
Graceful Fallback
Redis is entirely optional. If unavailable, caching and rate limiting are silently disabled and the API continues operating normally.
02

Features

Auth
JWT Authentication
Access and refresh token pair. Access tokens expire in 30 minutes; refresh tokens last 7 days and rotate on each use.
Access Control
Role-Based Access
Viewer, Analyst, and Admin roles enforced at the FastAPI dependency layer — no business-logic leakage.
Analytics
Dashboard Analytics
Income totals, expense totals, net balance, per-category breakdowns, and configurable monthly trend windows.
Performance
Redis Caching
Dashboard summary cached at dashboard:summary for 5 minutes. Invalidated automatically on any record write.
Safety
Soft Deletes
Financial records are flagged as deleted, not removed from the database — preserving audit history and enabling recovery.
Ops
Health Endpoint
A public /health endpoint returns service status for load balancers and uptime monitors.

Tech Stack

Each dependency was chosen for a specific role. The stack stays thin — no ORM magic, no heavy framework overhead.

FastAPI 0.110+
ASGI web framework. Handles routing, dependency injection, request validation, and Swagger/ReDoc doc generation automatically.
  • OpenAPI schema auto-generated from type hints
  • Async-native — compatible with async Redis and DB drivers
  • Global exception handlers for consistent error shapes
SQLAlchemy 2.0
ORM and query builder. Models defined as Python dataclasses — switching databases requires only a change to DATABASE_URL.
  • Works with SQLite (dev) and PostgreSQL (prod) out of the box
  • Tables created automatically on startup via init_db()
  • Session managed per-request through FastAPI dependency
Redis redis-py 5
Async key-value store powering dashboard response caching and per-IP rate limiting. Entirely optional — app degrades gracefully without it.
  • Dashboard summary cached with 5-minute TTL
  • Cache invalidated on every record write via pattern delete
  • Rate limiter uses sliding-window counters in Redis
JWT python-jose
Stateless authentication via signed tokens. Access tokens carry role claims for RBAC checks without a DB round-trip on every request.
  • HS256 algorithm, configurable to RS256 for multi-service setups
  • Access token: 30 min · Refresh token: 7 days
  • Refresh rotation — new pair issued on every refresh call
Pydantic v2 2.x
Schema validation and serialization for all request bodies and response models. Settings loaded via pydantic-settings from .env.
  • Strict type coercion at the boundary — no raw dict passing
  • BaseSettings handles env var loading and defaults
  • Cached with @lru_cache() for zero re-parse overhead
Uvicorn ASGI server
Lightweight ASGI server for running FastAPI in development and production. Supports hot reload and multi-worker deployments.
  • Development: --reload for live code reloading
  • Production: --workers N for multi-process scaling
  • Compatible with Gunicorn as a process manager

Redis — Caching & Rate Limiting

Redis serves two independent purposes in this API. Both are implemented as thin async wrappers in app/utils/redis_client.py and app/utils/rate_limiter.py.

Response Caching
01Request hits a dashboard endpoint
02Route handler calls cache_get(key)
03Cache hit → return JSON immediately (no DB query)
04Cache miss → query DB, call cache_set(key, data, ttl=300)
05On any record write → cache_delete_pattern("dashboard:*")
Cache Keys
dashboard:summary5 min TTL
dashboard:income5 min TTL
dashboard:expenses5 min TTL
dashboard:net_balance5 min TTL
dashboard:categories5 min TTL
dashboard:recent:{n}5 min TTL
dashboard:trends:{n}5 min TTL
ratelimit:{ip}sliding window counter
⚠ Graceful fallback: If Redis is unreachable on startup, get_redis() returns None. All cache_* and rate-limiter calls become silent no-ops — the API continues to function without caching or rate limiting. Configure REDIS_URL to enable both features.

Authentication

All auth endpoints are public — no token required. Use /login to obtain a token pair, and /refresh to rotate an expiring access token without re-entering credentials.

POST /api/v1/auth/login Public ▶

Authenticate with email and password. Returns an access token (30 min) and a refresh token (7 days).

Request Body

{
  "email": "admin@example.com",
  "password": "Admin@1234"
}

Response 200

{
  "access_token": "eyJhbGci...",
  "refresh_token": "eyJhbGci...",
  "token_type": "bearer"
}

Error 401

{ "detail": "Incorrect email or password." }
POST /api/v1/auth/refresh Public ▶

Exchange a valid refresh token for a new access and refresh token pair. The old refresh token is invalidated.

Request Body

{
  "refresh_token": "eyJhbGci..."
}

Response 200

{
  "access_token": "eyJhbGci...",
  "refresh_token": "eyJhbGci...",
  "token_type": "bearer"
}

Error 401

{ "detail": "Invalid or expired refresh token." }

Create a new financial record. Triggers cache invalidation for all dashboard:* keys.

  "type": "income",         // "income" | "expense"
  "category": "salary",
  "date": "2026-04-01",
  "notes": "March salary"   // optional
}

Authentication Flow

1

Login

POST to /api/v1/auth/login with email and password. You receive an access token (30 minutes) and a refresh token (7 days).

POST /api/v1/auth/login
{ "email": "admin@example.com", "password": "Admin@1234" }
2

Send the access token

Attach the token to every protected request via the Authorization header. The role encoded in the token determines what the request can access.

Authorization: Bearer <access_token>
3

Refresh when expired

When a 401 is returned, POST the refresh token to /api/v1/auth/refresh. A new token pair is issued without re-entering credentials. Store the new tokens and discard the old ones.

POST /api/v1/auth/refresh
{ "refresh_token": "eyJhbGci..." }
4

Handle rate limiting

Requests exceeding 60/minute per IP receive a 429 Too Many Requests response. Back off and retry after the window resets. Rate limiting is disabled if Redis is unavailable.

Project Setup

1

Clone the repository

git clone https://github.com/Harish-tig/Finance_Access_Control_Backend.git
cd Finance_Access_Control_Backend
2

Create and activate a virtual environment

python3 -m venv .venv
source .venv/bin/activate      # macOS / Linux
.venv\Scripts\activate         # Windows
3

Install dependencies

pip install -r requirements.txt
4

Configure environment

Copy the example env file and update the values. At minimum, set a strong SECRET_KEY before any production deployment.

cp .env.example .env
# then edit .env
SECRET_KEY=$(python3 -c "import secrets; print(secrets.token_hex(32))")
5

Start Redis (optional but recommended)

Enables dashboard response caching and per-IP rate limiting. The API runs without Redis — both features are silently disabled if the connection fails.

redis-server
# or via Docker:
docker run -d -p 6379:6379 redis:alpine
6

Run the server

uvicorn app.main:app --reload --host 0.0.0.0 --port 8000

On first boot, the database schema is created and the default admin account is seeded automatically. Swagger UI is available at http://localhost:8000/docs when DEBUG=true.

7

Verify the installation

# Health check
curl http://localhost:8000/health

# Login and get tokens
curl -X POST http://localhost:8000/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@example.com","password":"Admin@1234"}'
